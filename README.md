# CFO Operations Platform — Customer Deploy Guide

A Databricks Asset Bundle that deploys a CFO Operations Platform demo: synthetic financial data, a Genie space, two AI/BI dashboards, a Databricks App with chat UI and persona-keyed insight tiles, and a Claude-powered orchestrator that pre-computes root-cause analyses against the Genie space.

This guide takes you from `git clone` to a working demo running on **your** real data.

> **Bundle code lives in `cfo-app-light-working/`.** All `./deploy.sh ...` commands below assume you're inside that folder — `cd cfo-app-light-working` before running them. The sibling folder `cfo-app-code-clean/` is a legacy dark-theme reference; ignore unless you specifically need the dark-theme version.

---

## Architecture

```
[your-catalog].[your-schema]                  (your raw data)
   │  bronze views generated by customer_mapping.py
   ▼
[your-catalog].cfo_demo.bronze_*              (views in the shape our pipeline expects)
   │  silver/gold transformations from this bundle
   ▼
[your-catalog].cfo_demo.{silver_*, gold_*}    (tables built by the pipeline)
   │
   ▼
Genie space   ·   Dashboards   ·   App
```

No data movement — bronze is views over your existing tables. Silver/gold are materialized tables, refreshed by the bundle's pipeline.

## Prerequisites

### 1. Install the Databricks CLI

```bash
# macOS
brew tap databricks/tap
brew install databricks

# Linux / WSL
curl -fsSL https://raw.githubusercontent.com/databricks/setup-cli/main/install.sh | sh
```

Required version: `v0.299.1` or later. Verify:

```bash
databricks --version
```

### 2. Create a CLI profile for your workspace

```bash
databricks auth login --host https://<your-workspace>.cloud.databricks.com --profile <profile-name>
```

Pick any `<profile-name>` (e.g. `acme-prod`). Browser opens, you log in, profile saves to `~/.databrickscfg`. Verify:

```bash
databricks current-user me --profile <profile-name>
```

### 3. Required Unity Catalog permissions

Your user must have `CREATE SCHEMA` on the target catalog. Missing this permission is the most common first-deploy failure.

```sql
GRANT USE CATALOG, CREATE SCHEMA ON CATALOG <your-catalog> TO `<your-user-or-group>`;
```

Other permissions needed (workspace admin or metastore admin covers all):

- `USE SCHEMA`, `CREATE TABLE`, `MODIFY`, `SELECT` on the demo schema (auto-created)
- `CAN USE` on your SQL warehouse
- `CAN QUERY` on `databricks-claude-opus-4-7`, `databricks-claude-haiku-4-5`, `databricks-bge-large-en` model serving endpoints
- Create permissions for jobs, apps, dashboards, Genie spaces at the workspace level

### 4. Values to gather

| Value | How to find | Used as |
|---|---|---|
| Profile name | From step 2 above | `--profile <name>` |
| Catalog | Any UC catalog where you have `CREATE SCHEMA` | `--catalog <name>` |
| Schema | New schema name (bundle creates it). E.g. `cfo_proserv` | `--schema <name>` |
| SQL warehouse ID | Workspace UI → SQL Warehouses → click warehouse → copy ID from URL | `--warehouse-id <id>` |
| Genie space title | Any unique string. E.g. `"Acme CFO Demo"` | `--genie-space-title "<title>"` |

### 5. Workspace features

Modern Databricks workspaces have these enabled by default. Confirm with your SA if any are missing:

- AI Gateway + Foundation Model serving (Claude Opus 4.7, Haiku 4.5, BGE embeddings)
- Genie spaces
- AI/BI Dashboards (Lakeview)
- Databricks Apps

## Step 1 — Initial deploy (synthetic data)

From the repo root:

```bash
./deploy.sh \
  --profile <your-cli-profile> \
  --target dev \
  --catalog <your-catalog> \
  --schema cfo_proserv \
  --warehouse-id <your-warehouse-id> \
  --genie-space-title "<unique-title-for-this-deploy>"
```

What this does: creates the schema, hydrates synthetic bronze data, builds silver/gold tables, provisions the Genie space, deploys dashboards + app, then pre-caches insight tiles and chat answers.

Wall-clock: 5-7 min for the script. `generate_insights` continues for 25-30 min in the background — the app is usable immediately, chip clicks fall back to live agent generation (30-60s each) until the cache populates.

**Verify:** open the App URL printed at the end of the deploy. Click any chip — you should see a Claude response within seconds (cached) or minutes (live).

## Step 2 — Map your real schema

Open the customer-mapping notebook in the workspace UI:

```
/Workspace/Users/<you>/cfo-app/files/genie_insights/customer_mapping.py
```

Run it top-to-bottom:

1. **Set widgets** at the top. `CFO_CUSTOMER_SOURCES` = your real `catalog.schema`. For data spread across multiple catalogs (e.g. Workday in one catalog, SAP in another), use a semicolon-separated list:
   ```
   workday_catalog.hr_schema;sap_catalog.finance_schema;sfdc_catalog.crm_schema
   ```
   The notebook profiles the union and emits views that pull from each source catalog as needed.
2. **Run cells 1-6.** This profiles your schema, embeds columns, recall-matches against the bundle's expected bronze shape, has Claude rerank candidates, and writes `mappings.yaml` + `gaps.md` to workspace files.
3. **Review `mappings.yaml`.** For each column, confirm or correct the proposed source mapping. Cross-reference `gaps.md` for prioritized to-dos.
4. **Run the apply cell.** Set the `CFO_MAPPINGS_FILE` widget to the path of your edited YAML, then execute. This creates `bronze_*` VIEWS in `<your-catalog>.cfo_demo` that project your real data into the shape the bundle expects.

What you have after this: a new schema `<your-catalog>.cfo_demo` containing bronze views over your real data. Silver and gold tables do not exist yet — Step 3 builds them.

The synthetic schema from Step 1 (`cfo_proserv`) is **not touched** — those tables remain. After Step 3 succeeds, you can drop the synthetic schema manually if you want: `DROP SCHEMA <your-catalog>.cfo_proserv CASCADE;`. Keep it around if you'd like a fallback for debugging.

## Step 3 — Re-point the demo at your real data

```bash
./deploy.sh \
  --profile <your-cli-profile> \
  --target dev \
  --catalog <your-catalog> \
  --schema cfo_demo \
  --warehouse-id <your-warehouse-id> \
  --genie-space-id <existing-space-id> \
  --genie-space-title "<same-title-as-step-1>" \
  --refresh-data \
  --skip-bronze-hydrate
```

Two new flags:

- `--skip-bronze-hydrate`: the bundle's bronze tables don't exist in `cfo_demo` — your views from Step 2 are the bronze layer. This flag tells the pipeline to skip bronze generation.
- `--refresh-data`: triggers `cfo_data_pipeline` to rebuild silver and gold tables on top of your bronze views, then regenerates the chip pre-cache against your real data.

Find `--genie-space-id` in the workspace UI: Genie → your space → URL contains the ID. (Omitting it also works — the script falls back to title lookup.)

Wall-clock: 30 min. App, dashboards, and Genie now point at `<your-catalog>.cfo_demo` and serve your real data.

---

## Step 4 (optional) — Schedule recurring refreshes

If your underlying data updates regularly, schedule the `cfo_data_pipeline` job so silver/gold + the chip pre-cache stay fresh. In `databricks.yml`, find the job definition and add a `schedule` block. Or set it through the UI: Workflows → `cfo_data_pipeline` → Schedule.

## Re-deploy reference

| What changed | Command |
|---|---|
| App code, prompts, instructions, dashboards | Same deploy command as Step 3, **without** `--refresh-data` and **without** `--skip-bronze-hydrate` |
| Data layer (silver/gold + chip cache) | Same deploy command as Step 3, **with** `--refresh-data` and `--skip-bronze-hydrate` |
| Repoint same `--target` to a different workspace | `rm -rf .databricks/bundle/<target>` first, then deploy (clears stale local terraform state) |

## Troubleshooting

**`PERMISSION_DENIED: User does not have CREATE SCHEMA on Catalog '...'`**
Run the `GRANT` from Prerequisites §3, or deploy to a catalog you own.

**`Unable to find dashboard [...]`** during `[3/6] Bundle deploy`
Local terraform state has IDs from a different workspace. Clear it:
```bash
rm -rf .databricks/bundle/<target>
```
Then re-run the deploy.

**Genie space lookup fails on re-deploy**
The `--genie-space-title` doesn't match an existing space. Either pass the explicit `--genie-space-id` or use the exact title from your initial deploy.

**Apps deploy fails with "maximum limit of 200 apps"**
Workspace cap reached. Delete unused apps in the UI or ask your admin to raise the cap.

## `deploy.sh` flag reference

| Flag | Required | Notes |
|---|---|---|
| `--profile` | Yes | CLI profile name. |
| `--catalog` | Yes | Unity Catalog catalog. |
| `--schema` | Yes | Schema where demo data lives. |
| `--warehouse-id` | Yes | SQL warehouse ID. |
| `--target` | No | Bundle target (`dev` or `prod`). Default: `dev`. |
| `--genie-space-title` | No | Display title for the Genie space. Make it unique per deploy. |
| `--genie-space-id` | No | Use on re-deploy to skip title lookup. |
| `--app-name` | No | App name in workspace. Default: `cfo-demo`. |
| `--workspace-root-path` | No | Bundle sync root. Default: `/Workspace/Users/<you>/cfo-app`. |
| `--refresh-data` | No | Runs `cfo_data_pipeline` at the end. Adds 25-30 min. Required for Step 3. Not needed on first-time deploy (auto-triggered). |
| `--skip-bronze-hydrate` | No | Pipeline skips bronze generation. Required for Step 3 (your views are bronze). |
